# 06 — Repository Ports & Adapters

**Slice:** Write/query ports + adapters + CookieModel
**Reviewer:** ddd-specialist
**Date:** 2026-05-22
**Source files reviewed:** 7

## TL;DR
Hexagonal separation is genuinely clean — write port (`Ports/CookieRepositoryInterface`) and read port (`Ports/CookieQueryRepositoryInterface`) are domain-pure (entity / DTO only, no CI4 types), and the adapters live under `Repositories/`. Optimistic locking, outbox, tenant stamping, audit cols, and a duplicate-key catch are wired. But the template still ships landmines a `sed s/Cookie/Foo/g` developer will inherit verbatim: hard-coded `'cookie'` metric slice key, the `Models/Cookie/Traits/*` namespace that any naive rename leaves broken, divergent soft-delete-name semantics between `existsByName` (rejects trashed) and the schema (allows reuse), a `LIKE` search with no wildcard escape, and read/write paginate using opposite default sort orders.

## Verdict
READY-WITH-FIXES

## Findings

### F1 — HIGH — `existsByName` contradicts the schema's reuse-after-soft-delete contract
- **Location:** `app/Models/Cookie/CookieModel.php:93-115`; migration `2025-01-21-000001_CreateCookiesTable.php:25-28,128-130`
- **Observation:** `existsByName`/`existsByNameExcludingId` call `withDeleted()`, so the handler-side check refuses a name that matches a soft-deleted row. The migration docblock explicitly says the composite `UNIQUE(tenant_id, name, deleted_at)` exists precisely so "soft-deleted rows do not block creation of a new row with the same name (B16), restoration is possible (B17)". Handler is stricter than schema.
- **Why this is a template defect:** Every cloned domain inherits a name-uniqueness rule that disagrees with its own DB index. The first time someone soft-deletes "Foo" they can never create a new "Foo", contradicting the documented ERP semantics that the Cookie domain is supposed to model.
- **Suggested fix:** Drop `withDeleted()` from both `existsByName*` methods (or split the check: "exists live" vs "exists including trashed", and have the handler call the live variant for create, the trashed variant for restore-conflict detection).

### F2 — HIGH — Hard-coded `'cookie'` metric slice key survives sed
- **Location:** `app/Models/Cookie/Traits/BusinessMetricsLogging.php:47,54`
- **Observation:** `$this->loggingConfig->metricsThresholds['cookie'] ?? []` is keyed on the lowercase domain string. A developer running `sed s/Cookie/Foo/g` renames classes and namespaces, but the literal lowercase `'cookie'` inside the trait will not be touched (case differs; the trait file itself is also under `app/Models/Cookie/Traits/` so a naive directory rename misses any literal). Result: cloned Foo domain reads metric thresholds from the wrong config slice and silently falls back to the 10-units / 10% defaults.
- **Why this is a template defect:** Hidden coupling between domain identifier and config key. Either auto-derive the key from `static::class`, accept it as a trait constructor param, or document a manual touch-up step in the scaffolder.
- **Suggested fix:** Make the trait abstract on a `protected string $metricsSliceKey` and force the consumer to declare it, or derive from a `LogContext::DOMAIN` constant pattern used elsewhere.

### F3 — HIGH — `BusinessMetricsLogging` / `RepositoryLogging` traits hard-code `'Cookie'` / `'CookieRepository'` literal strings
- **Location:** `app/Models/Cookie/Traits/RepositoryLogging.php:25-58,75-81,104-114`; `BusinessMetricsLogging.php:72-78,104-112,135-141`
- **Observation:** Every log payload writes `'domain' => 'Cookie'` and `'repository' => 'CookieRepository'` as string literals. Cloning the trait into a new namespace requires hand-editing each entry. The PSR-3 channel already carries the class name; the duplication is pure rename risk.
- **Why this is a template defect:** Sed-clone produces log entries that misattribute production failures to the Cookie domain. Worse, error codes (`ErrorCodes::COOKIE_REPOSITORY_SAVE_FAILED`) are also hard-wired; observability dashboards filtering on those error codes will silently miss the cloned domain.
- **Suggested fix:** Either accept `$domain` / `$repository` as constructor-injected scalars on the repository (passed into the trait via property), or replace the traits with thin per-domain logger classes generated by the scaffold.

### F4 — HIGH — Read repository unescaped `LIKE` allows `%` / `_` injection in search term
- **Location:** `app/Domain/Cookie/Repositories/CookieQueryRepository.php:131`; mirror in `CookieRepository.php:554`
- **Observation:** `$builder->like('name', $searchTerm)` passes user input directly to a SQL `LIKE`. CI4's `like()` does not escape wildcards. A search for `100%` matches every row whose name contains `100` followed by anything; a search for `_` matches any single character. Worse, on large tables a leading-wildcard `LIKE '%term%'` (CI4 default) cannot use an index → unintended table scan when malicious / pathological input arrives.
- **Why this is a template defect:** This is the search recipe every cloned domain will copy. Pattern-special chars from end-user input are a documented CI4 footgun.
- **Suggested fix:** Escape `%` and `_` (`strtr($term, ['%' => '\\%', '_' => '\\_'])`) before passing to `like()`, and document the wildcard side (`'after'`/`'both'`) explicitly.

### F5 — MEDIUM — Write paginate orders by `created_at DESC`, read paginate orders by `id ASC` — inconsistent UX
- **Location:** `CookieRepository.php:568` (write side `executeFindPaginated`) vs `CookieQueryRepository.php:138` (read side)
- **Observation:** Same `findPaginated` port signature on two ports, two opposite implicit sorts. A query handler reading via the read repo sees the oldest cookies first; the write-side `findPaginated` (called from internal contexts) returns newest first.
- **Why this is a template defect:** Symmetric ports must have symmetric semantics. The whole point of the CQRS split is that the read port is authoritative for queries — and yet the write repo's `findPaginated` exists too (interface declares it). Either remove `findPaginated` from `CookieRepositoryInterface` (write side has no business paginating, that's a read concern) or align the order-by clauses.
- **Suggested fix:** Remove `findAll` + `findPaginated` from `CookieRepositoryInterface` (they're read methods on a write port). If you keep them for internal use, share the ORDER BY constant.

### F6 — MEDIUM — `CookieModel::existsByName` uses `LOWER(name)` which voids the index
- **Location:** `app/Models/Cookie/CookieModel.php:96,112`
- **Observation:** `where('LOWER(name)', strtolower($name))` forces a full table scan: MySQL cannot use the `(tenant_id, name, deleted_at)` UNIQUE index when the leftmost predicate function-wraps the column. The migration's docblock claims case-insensitive uniqueness is achieved via the pinned `utf8mb4_unicode_ci` collation (`migration:62-63`) — so a plain `where('name', $name)` would use the index AND be CI. The `LOWER()` wrapper is belt-and-suspenders that costs an index.
- **Why this is a template defect:** Every cloned domain inherits a slow uniqueness check that grows linearly with table size.
- **Suggested fix:** Drop `LOWER()`. Rely on the `utf8mb4_unicode_ci` collation that the migration explicitly pins for this reason.

### F7 — MEDIUM — Reconstitute path reads VOs by re-validating — a corrupt row throws on read
- **Location:** `CookieRepository.php:373-387` (`toDomainEntity`)
- **Observation:** `CookieName::fromString((string) $data['name'])` and `CookiePrice::fromString((string) $data['price'])` re-run the full VO validators on every row pulled from the DB. A name that was 100 chars at insert time but is now in a column some migration shrank to 50, or a price stored as `'0.00'` from a legacy import, will throw `ValidationException` mid-list. The read repository (`CookieQueryRepository::formatPrice`) defensively catches this; the write repo's `toDomainEntity` does not.
- **Why this is a template defect:** Trust boundaries. Data already in the DB has been validated once; reconstituting an aggregate should not be capable of failing on data-shape grounds, otherwise a single bad row poisons every paginated list and every `findById`. Most DDD references use a separate "trusted constructor" path for reconstitution.
- **Suggested fix:** Add `CookieName::fromTrusted(string)` / `CookiePrice::fromTrusted(string)` that skip validation, or have `reconstitute` accept the raw scalar and never round-trip through `fromString`.

### F8 — MEDIUM — `delete()` does a SELECT before the soft-delete (N+1 of one)
- **Location:** `CookieRepository.php:294-318`
- **Observation:** `$cookie = $this->findById($id)` runs a full row fetch + VO reconstitution + popular-cookie tracking, just to return `false` if missing or to know whether to call `model->delete($id)`. Then another UPDATE for `deleted_by` and another for the soft-delete itself. Three round-trips for one delete.
- **Why this is a template defect:** The pattern propagates to every cloned domain. A single UPDATE `WHERE id = ? AND deleted_at IS NULL SET deleted_at = NOW(), deleted_by = ?` would do the job in one trip and return `affected_rows` for the existence check.
- **Suggested fix:** Replace the SELECT-then-UPDATE with a single conditional UPDATE; check `affectedRows` for the boolean return.

### F9 — MEDIUM — `restore()` returns the raw `update()` truthy result, ignoring `affectedRows()`
- **Location:** `CookieRepository.php:343-345`
- **Observation:** `return $this->model->builder()->where('id', $id)->update($update);` — CI4's QueryBuilder `update()` returns `true` on a successful statement *even when zero rows matched*. The caller can't tell whether the row was restored or whether someone hard-deleted it between the `findByIdWithTrashed` check and the update. Also, restore bypasses optimistic locking entirely (no `version` increment, no `WHERE version = ?`).
- **Why this is a template defect:** Restore is a state mutation; it should bump version like every other write. As-is, an in-flight stale `Cookie` entity will silently overwrite a restored row.
- **Suggested fix:** Bump `version` in the UPDATE payload + `WHERE` clause; verify via `affectedRows() === 1`.

### F10 — MEDIUM — `CookieQueryRepository::findPaginated` swallows a false `get()` result silently
- **Location:** `CookieQueryRepository.php:140-142`
- **Observation:** `$rows = $result === false ? [] : $result->getResultArray();` — if the data fetch fails after the count succeeded, the caller gets `total > 0` and `data === []`. The write side correctly throws `RuntimeException` in this case (`CookieRepository.php:570-572`). Asymmetric error handling between the two ports.
- **Why this is a template defect:** Read-side query failures vanish into empty pages; observability and UX both suffer.
- **Suggested fix:** Mirror the write side — throw on `get() === false`.

### F11 — LOW — `CookieModel` is not `final`, and its parent `CodeIgniter\Model` exposes a public `$db`
- **Location:** `CookieModel.php:28`; usage at `CookieRepository.php:475,492`
- **Observation:** The repository reaches into `$this->model->db->affectedRows()` and `$this->model->builder()` — two leaky abstractions the repo is forced to use because `Model` doesn't expose them via a clean interface. `CookieModel` is also not `final`, contradicting the project rule "Final classes by default" (CLAUDE.md).
- **Why this is a template defect:** Cloned domains inherit the same leak. Refactor candidate: wrap `affectedRows` and `builder()` access in named methods on `CookieModel` so the repository talks to its own thin facade.
- **Suggested fix:** `final class CookieModel`, expose `lastAffected(): int` and `queryBuilder(): BaseBuilder` wrapper methods.

### F12 — LOW — `CookieRepository::findById` triggers `trackPopularCookie` — read effect on a write-side adapter
- **Location:** `CookieRepository.php:208`
- **Observation:** The write-side `findById` (used by the save / delete / restore preloads above) increments the popularity counter on every lookup. So every UPDATE that reads the current version to compute "old price" bumps the popularity stat by one, polluting the metric.
- **Why this is a template defect:** Mixes read-side concern (popularity) into write-side adapter. The read repo doesn't have this hook at all.
- **Suggested fix:** Move popularity tracking to the read repository's `findById`, where it belongs.

### F13 — LOW — `CookieRepositoryInterface::save` returns `int` but write-on-insert mutates the entity (`assignId`) — undocumented side effect
- **Location:** `CookieRepository.php:449-452`; interface at `CookieRepositoryInterface.php:28`
- **Observation:** `save()` declares `int` return for the id, but it also mutates the caller's `Cookie` entity (`bumpVersion()` and `assignId()`). The port docblock doesn't mention this. A naive caller who reads the docblock and discards the return value still mutates their `$cookie` object.
- **Why this is a template defect:** Hidden side effects on a "port" are exactly the surprise template users hit when they assume `save($cookie)` is value-only.
- **Suggested fix:** Document in the interface docblock that `save` mutates the entity (assigns id on insert, bumps version on every persist). Optionally split into `save()` for insert returning a new Cookie and `update()` for the lock path.

### F14 — LOW — `CookieRepositoryInterface` declares only `delete` (soft) but no hard-delete escape hatch
- **Location:** `CookieRepositoryInterface.php:63`
- **Observation:** The port exposes a soft-delete and a restore but no hard-delete. For GDPR / right-to-erasure cases (which an ERP template almost certainly needs), there's no way to truly purge a row through the port; callers would have to reach into `CookieModel` directly, breaking the port abstraction.
- **Why this is a template defect:** A reference template should anticipate the hard-delete-for-compliance case and either declare it explicitly forbidden or expose `purge(int $id): bool`.
- **Suggested fix:** Add a `purge()` method or document why it is intentionally absent.

### F15 — LOW — `BaseConnection` template parameter in `CookieQueryRepository` constructor is over-specified
- **Location:** `CookieQueryRepository.php:44,52`
- **Observation:** `BaseConnection<object|resource|false, object|resource|false>` — the generic template makes the constructor signature exotic. Tests injecting a mock DB connection now need to align the template, which is fiddly. The write side never bothers with this template parameter.
- **Why this is a template defect:** Asymmetric DX between the two adapters; the read repo is harder to fake than the write repo.
- **Suggested fix:** Drop the generic params and accept `BaseConnection` plain, or introduce a `DatabaseConnectionInterface` to ditch the CI4 template entirely.

### F16 — INFO — `CookieModel::$validationRules` duplicates VO invariants
- **Location:** `CookieModel.php:56-79`
- **Observation:** The model declares `min_length[3]`, `max_length[100]`, `greater_than[0]`, `greater_than_equal_to[0]` — exactly the invariants `CookieName` / `CookiePrice` / `CookieStock` already enforce. Either VO validation is sufficient (and the model rules are dead weight) or DB-level validation is sufficient (and the VOs are pure conversion). Currently we pay for both.
- **Why this is a template defect:** When a developer changes the VO's min length to 2 but forgets the model, the save silently fails with a CI4 validation message instead of a VO-level `ValidationException`. The model layer's validation runs *after* the VO has already accepted the value.
- **Suggested fix:** Pick one validation layer (the VO) and document the model's rules as DB-shape safety only (types/required, no business ranges).

### F17 — INFO — Write port declares `existsByName` / `existsByNameExcludingId` as `string` — leaks raw scalar
- **Location:** `CookieRepositoryInterface.php:53,58`
- **Observation:** These should accept `CookieName`. As-is, the handler must unwrap the VO to call the port (or worse, can call the port with un-validated user input). Port hygiene = VOs in, VOs out.
- **Why this is a template defect:** Cloned domains inherit a port that bypasses its own value-object boundary.
- **Suggested fix:** `existsByName(CookieName $name): bool;` and `existsByNameExcludingId(CookieName $name, int $excludeId): bool;`.

## What is correct / praiseworthy
- Ports under `Domain/Cookie/Ports/` are CI4-free: only entity / DTO / Actor VO. Hexagonal placement is exemplary.
- Write port stamps `Actor` for `created_by` / `updated_by` / `deleted_by` and routes it cleanly without leaking actor into the entity.
- Optimistic locking is implemented end-to-end: `version` bumped in `WHERE`, conflict raises a domain-level `concurrentModification` with the actual + expected version.
- Outbox-first / dispatcher-second ordering in `dispatchPendingEvents()` is the correct durability semantic (event commits with the row).
- `CookieQueryRepository` is genuinely read-only (DTOs, no entity), tenant-filtered, with a defensive `formatPrice` that doesn't crash on malformed source data.
- Duplicate-key catch in `save()` translates SQL exceptions into domain `businessRuleViolation` with a stable error code.
- Pagination clamping (`max(1, min(100, $perPage))`) on the read side prevents accidental full-table dumps.
- `assignId` refuses to overwrite an existing id — defensive against double-save.
- `findByIdWithTrashed` correctly named (`WithTrashed` is the documented Laravel-ism every ORM hits) and used only by the restore path.

## Top 3 fixes before cloning
1. **Reconcile `existsByName` with the schema's reuse-after-delete contract (F1).** Either drop `withDeleted()` from the model or change the migration's UNIQUE definition. Today they disagree and the docblock lies.
2. **De-hard-code the `'cookie'` metric slice key and the `'domain' => 'Cookie'` log fields out of the traits (F2, F3).** They survive sed and silently misattribute metrics + logs in every cloned domain.
3. **Trusted reconstitution path (F7) + escape `LIKE` wildcards (F4).** Read paths that throw on legacy rows, or that accept `%` as user input, are the kind of latent production bug every cloned domain will inherit verbatim.
